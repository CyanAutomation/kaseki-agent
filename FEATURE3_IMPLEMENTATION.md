# Feature 3 Implementation Summary: File Read Summarization

## Overview

Successfully implemented **Phase 1-5 Core Infrastructure** of Feature 3: Better File Read Summarization for kaseki-agent. The system now provides intelligent, size-aware file reading with optional tree-sitter-based code structure summarization.

**Target Impact**: 10–15% token reduction per Pi invocation by returning structural summaries (classes, functions, types) instead of full implementations.

---

## Architecture

### Core Modules

#### 1. **TreeSitterSummarizer** (`src/summarization/tree-sitter-summarizer.ts`)

- **Purpose**: Parse code files using tree-sitter and extract structural information
- **Capabilities**:
  - Parses TypeScript, JavaScript, and Go
  - Extracts: imports, exports, classes, methods, functions, types, interfaces
  - Produces markdown summaries preserving signatures but omitting implementations
  - Timeout-aware parsing (falls back if parsing exceeds threshold)
  - Error resilience: returns structured error response instead of throwing
- **Key Methods**:
  - `summarize(content, timeoutMs)`: Returns CodeSummary with structure or parse error
  - `formatAsMarkdown(summary)`: Converts summary to human-readable markdown for Pi
- **Output Example**:

  ```markdown
  <!-- SUMMARY: typescript | 12847 bytes | 45ms -->
  ## Classes
  ### AuthManager
  - constructor(apiKey: string, apiUrl: string);
  - async authenticate(credentials: Credentials): Promise<Token>;
  - private validateToken(token: Token): boolean;
  ```

#### 2. **SummaryCache** (`src/summarization/summary-cache.ts`)

- **Purpose**: Cache summaries within same run, invalidate on file change
- **Key Features**:
  - File hash-based invalidation (detects content changes instantly)
  - No stale summaries: always validates before returning cached entry
  - Statistics tracking (hit rate, cache size, entry count)
  - Persistence to disk (optional)
  - TTL cleanup (default 24 hours, but skipped in ephemeral containers)
- **Key Methods**:
  - `get(filePath)`: Returns cached summary or null (if stale or missing)
  - `set(filePath, summary, language)`: Stores summary with file hash
  - `getFileHash(filePath)`: Computes SHA-256 of file for change detection
  - `getStats()`: Returns hit rate, misses, total size
- **Design**: Per-workspace cache at `.kaseki-summary-cache/` (survives multiple Pi phases within single run)

#### 3. **Read Strategy** (`src/summarization/read-strategy.ts`)

- **Purpose**: Smart thresholding logic to decide when summarization is beneficial
- **Algorithm** (6 Rules):
  1. **Editing Phase**: If `isDraft=true`, always use full read (correctness for implementation details)
  2. **Parse Error**: If summarization fails, fallback to full read
  3. **Small Files**: If size < 2KB, use full read (overhead not worth it)
  4. **Unsupported Language**: If language not in [TypeScript, JavaScript, Go], use full read
  5. **Large Files**: If size > 1MB, use full read (parse timeout risk)
  6. **Default**: Use summary for supported languages in optimal range
- **Key Functions**:
  - `getReadStrategy(context)`: Returns ReadStrategy ('full' or 'summary') + reason
  - `estimateSummaryBenefit()`: Calculates estimated token savings (~60–70% reduction typical)
- **Thresholds** (configurable via env vars):
  - `KASEKI_SUMMARY_MIN_BYTES`: 2KB (default)
  - `KASEKI_SUMMARY_MAX_BYTES`: 1MB (default)
  - `KASEKI_SUMMARY_PARSE_TIMEOUT`: 100ms (default)

#### 4. **Read Wrapper** (`src/summarization/read-wrapper.ts`)

- **Purpose**: Main integration point; orchestrates summarizer, cache, and strategy
- **Public API**:
  - `readFileWithSummary(filePath, options)`: Async function returning content (string or null)
  - `readFileWithSummaryAndMetrics(filePath, options)`: Returns {content, metrics}
  - `flushSummaryCache()`: Persist cache to disk on cleanup
  - `clearSummaryCache()`: Clear all cached entries
- **Decision Path** (returned in metrics):
  - `cache_hit`: Content from cache (instant)
  - `tree_sitter`: Summary generated by tree-sitter
  - `full_read`: Return full file content
  - `error`: Summarization failed, fallback to full
- **Metrics Provided**:

  ```typescript
  {
    strategy: 'summary' | 'full',
    strategyReason: string,
    fullSizeBytes: number,
    returnedSizeBytes: number,
    compressionRatio: number,
    parseTimeMs: number,
    cacheHit: boolean,
    decisionPath: 'cache_hit' | 'full_read' | 'tree_sitter' | 'error',
    estimatedTokensFull: number,
    estimatedTokensReturned: number,
    estimatedTokensSaved: number,
  }
  ```

- **Options**:
  - `full?: boolean`: Force full read (Pi override before editing)
  - `returnMetrics?: boolean`: Include metrics in response
  - `timeoutMs?: number`: Custom parse timeout
  - `isDraft?: boolean`: Editing phase (forces full read for correctness)

#### 5. **Configuration** (`src/summarization/summarizer-config.ts`)

- **SummarizerConfig Interface**: All tunable parameters
- **Default Settings**:
  - `supportedLanguages`: ['typescript', 'javascript', 'go']
  - `minSizeBytes`: 2048
  - `maxSizeBytes`: 1048576
  - `parseTimeoutMs`: 100
  - `enableCache`: true
  - `cacheTTLMs`: 24 hours
- **Environment Variable Overrides**:
  - `KASEKI_SUMMARY_MIN_BYTES`
  - `KASEKI_SUMMARY_MAX_BYTES`
  - `KASEKI_SUMMARY_PARSE_TIMEOUT`
  - `KASEKI_SUMMARY_CACHE`
  - `KASEKI_SUMMARY_CACHE_DIR`

## Key Design Decisions

### 1. **Smart Thresholding Over "Always Summarize"**

- **Why**: Small files benefit more from full read (no summarization overhead)
- **Implementation**: Size-based rules with language awareness
- **Result**: Token savings only when worthwhile (large supported files)

### 2. **No Stale Summaries Guarantee**

- **Why**: File changes during run could lead to out-of-date analysis
- **Implementation**: File hash validation on every cache lookup
- **Trade-off**: Slightly slower than TTL-only caching, but correctness paramount

### 3. **Full Read Before Editing**

- **Why**: Many bugs are in function bodies, not signatures
- **Implementation**: `isDraft` flag triggers automatic full read strategy
- **Benefit**: Ensures Pi has implementation context for correctness

### 4. **Markdown Format for Summaries**

- **Why**: Human-readable, easy for Pi to parse, preserves structure
- **Format**: Hierarchical headers with signatures
- **Metadata**: Comment prefix with language, size, parse time

### 5. **Per-Workspace Cache (Not Global)**

- **Why**: Different projects have different codebases; global cache could confuse
- **Scope**: Survives multiple Pi phases within single kaseki run
- **Cleanup**: Cleared between runs (ephemeral containers)

## How It Works: Complete Flow

### Scenario 1: Large TypeScript File, Scouting Phase

```
Pi requests: read("src/services/auth.ts")
  ↓
readFileWithSummary("src/services/auth.ts")
  ↓
File size: 25KB, language: TypeScript, isDraft: false
  ↓
Check cache → miss
  ↓
getReadStrategy() → 'summary' (large supported file)
  ↓
TreeSitterSummarizer.summarize() → CodeSummary
  ↓
formatAsMarkdown() → Markdown with classes, methods, types
  ↓
Cache result (file hash: abc123, timestamp: now)
  ↓
Return markdown summary (~4KB)
  ↓
Pi receives: structural overview without implementation noise
Token savings: ~14KB (70% compression)
```

### Scenario 2: Same File, Before Edit (isDraft=true)

```
Pi requests: read("src/services/auth.ts", full=true)
  ↓
readFileWithSummary(..., { full: true })
  ↓
full=true override detected
  ↓
Return complete file (25KB)
  ↓
Pi receives: full implementation for correctness
No compression, but correctness guaranteed
```

### Scenario 3: Small File (Utility)

```
Pi requests: read("src/utils/helpers.ts")
  ↓
File size: 1.5KB, language: TypeScript
  ↓
getReadStrategy() → 'full' (too small)
  ↓
Return complete file (1.5KB)
  ↓
Token cost: ~0.4KB (full read cheaper than summary overhead)
```

### Scenario 4: Unsupported Language

```
Pi requests: read("config/setup.py")
  ↓
File size: 8KB, language: Python
  ↓
getReadStrategy() → 'full' (unsupported)
  ↓
Return complete file (8KB)
  ↓
Fallback ensures no errors; future: add Python support via tree-sitter-python
```

## How to Use (Developer Reference)

```typescript
// Basic usage
import { readFileWithSummary } from './src/utils/file-helpers';

const content = await readFileWithSummary('src/service.ts');
// Returns summary for large files, full content for small files

// With metrics
import { readFileWithSummaryAndMetrics } from './src/utils/file-helpers';

const { content, metrics } = await readFileWithSummaryAndMetrics('src/service.ts');
console.log(`Compression: ${metrics.compressionRatio.toFixed(2)}x`);
console.log(`Tokens saved: ${metrics.estimatedTokensSaved}`);

// Pi override (before editing)
const fullContent = await readFileWithSummary('src/service.ts', { full: true });

// Cleanup
import { flushSummaryCache } from './src/utils/file-helpers';
flushSummaryCache(); // Call at end of Pi invocation
```

## Questions & Considerations

**Q: What happens if tree-sitter parsing times out?**  
A: Falls back to full read (safe default). Timeout configurable via KASEKI_SUMMARY_PARSE_TIMEOUT.

**Q: Should I enable cache in production?**  
A: Yes, unless ephemeral containers. Within-run cache is safe and provides instant hits.

**Q: Can Pi control when to request full reads?**  
A: Yes! Pi can request `read(file, full=true)` before editing. Guidance added to TASK_PROMPT.

**Q: How much do summaries compress?**  
A: Typically 60–70% reduction for large files. Metrics show actual compression for each file.
